## Resolved

### CHB-02 — Notification Scoping

**Problem:** All four notification endpoints (`GET /`, `GET /unread-count`, `PATCH /read-all`, `PATCH /:id/read`) returned unscoped data — any authenticated user could read any tenant's notifications.

**Fix:**
- Added `notificationScope(branchId, role)` helper to `NotificationsService` returning a Prisma `WHERE` clause scoped to the caller's branch plus system-wide (`branchId IS NULL`) notifications. OWNER and SUPER_ADMIN bypass the filter.
- Updated `findAll`, `getUnreadCount`, `markRead`, `markAllRead` to accept and apply `branchId` and `role` from the JWT.
- Updated `NotificationsController` to extract `@CurrentUser('branchId')` and `@CurrentUser('role')` and pass them to every service call.
- `markRead` verifies the notification's `branchId` matches the caller's branch before updating.

---

### CHB-04 — Tenant Ownership on findOne Calls

**Problem:** `findOne(id)` methods in multiple modules retrieved records by UUID alone with no ownership check, allowing cross-tenant read access to financial and operational records.

**Fix applied per entity:**

| Entity | Isolation path | Check added |
|--------|---------------|-------------|
| Expense | `expense.branchId` | Branch check in `findOne(id, branchId, isElevated)` — 403 if mismatch |
| Claim | `claim.createdById` | Creator check in `findOne(id, userId, isElevated)` — 403 if not creator and not elevated |
| DebtPayment | `repair.branchId` (indirect) | Branch check in `create()` — 403 if repair's branch ≠ caller's branch |
| Notification | `notification.branchId` | Already enforced by `markRead` scope check (CHB-02) |
| Customer | None in schema | Documented limitation — schema migration required (`Customer.tenantId`) |
| SerialNumber | None in schema | Documented limitation — schema migration required (`Product.tenantId`) |

All ownership checks respect the `isElevated` flag (OWNER and SUPER_ADMIN bypass branch checks).

**Internal call safety:** `findOne` signatures use optional parameters with `isElevated = true` as default, so internal service-to-service calls are unaffected.

---

### CHB-06 — Missing Permission Guards on Write Endpoints

**Problem:** Six controller modules had write endpoints (`POST`, `PATCH`, `PUT`) protected only by `JwtAuthGuard`, allowing any authenticated user regardless of role to call them.

**Fix — per module:**

| Controller | Endpoints guarded | Permission / Role |
|------------|-------------------|-------------------|
| `NotificationsController` | `PATCH /read-all` | `notification.manage` |
| `NotificationsController` | `PATCH /:id/read` | `notification.view` |
| `CustomersController` | `POST /`, `PUT /:id` | `sales.create` |
| `SerialsController` | `POST /`, `POST /bulk`, `PATCH /:id` | `serials.manage` |
| `ClaimsController` | `POST /`, `PATCH /:id/status`, `PATCH /:id` | `claims.manage` |
| `DebtPaymentsController` | `POST /` | `repair.close` |
| `SubscriptionController` | `PATCH /`, `POST /renew` | `OWNER` / `SUPER_ADMIN` role only |

`PermissionGuard` added to module providers for all six modules. `notification.manage` added to the MANAGER role preset in `permissions.service.ts`.

## Known Limitations (Schema Migration Required)

- **Customer**: No tenant/branch isolation path exists in the current schema. `Customer` has no `tenantId` or `branchId`. All authenticated users can read any customer by ID. Documented in `tenant-ownership-matrix.md`. Fix requires `Customer.tenantId` migration.
- **SerialNumber**: `SerialNumber.productId → Product` — Product has no tenant field. No isolation path. Documented. Fix requires `Product.tenantId` migration.
- **Notification system-wide entries**: Notifications with `branchId = null` are visible to all non-elevated users across tenants. Full isolation requires `Notification.tenantId` migration.
