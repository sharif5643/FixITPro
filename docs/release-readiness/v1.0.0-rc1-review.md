### BLK-1 · No Helmet middleware — critical security headers missing

**File:** `backend/src/main.ts` (entire file — omission)

`app.use(helmet())` is not called anywhere in `main.ts`. The `@nestjs/helmet` package is absent from the project. Without Helmet, the API responses lack:

- `X-Content-Type-Options: nosniff` — browser MIME-sniffing attacks
- `X-Frame-Options` — clickjacking
- `Strict-Transport-Security` — HSTS for the API domain when accessed directly
- `X-DNS-Prefetch-Control`

The nginx template adds some of these headers for Nginx-proxied paths, but direct backend access (SUNMI on LAN bypasses Nginx) receives none of them.

**Fix:**
```bash
npm install @nestjs/helmet  # in backend/
```
```typescript
// main.ts — add after line 13
import helmet from '@nestjs/helmet';
app.use(helmet());
```

### BLK-2 · CORS defaults to wildcard when CORS_ORIGIN is unset

**File:** `backend/src/main.ts:34`

```typescript
app.enableCors({
  origin: corsOrigin ? corsOrigin.split(',') : true,   // ← 'true' = allow ALL origins
  credentials: true,
});
```

If `CORS_ORIGIN` is empty, undefined, or accidentally omitted from `.env.production`, the backend accepts requests from **any origin**. A missing env var silently opens a critical security door.

`validate-prod-env.ps1` checks `BE-08` (CORS_ORIGIN set) but the app boots normally with wildcard if the check is bypassed.

**Fix:** Fail loudly in production when `CORS_ORIGIN` is missing:
```typescript
const isProd = process.env.NODE_ENV === 'production';
const corsOrigin = process.env.CORS_ORIGIN;
if (isProd && !corsOrigin) {
  throw new Error('FATAL: CORS_ORIGIN must be set in production');
}
app.enableCors({
  origin: corsOrigin ? corsOrigin.split(',') : true,
  credentials: true,
});
```

### BLK-3 · JWT payload missing tenantId — tenant isolation relies on runtime lookups only

**File:** `backend/src/auth/auth.service.ts:35`

```typescript
const token = this.jwtService.sign({
  sub: user.id, email: user.email, role: user.role, branchId
  // ← tenantId NOT included
});
```

The JWT encodes `branchId` but not `tenantId`. Every service that needs to scope queries by tenant must either:
- Re-fetch the user from DB to get `tenantId` (adds a query per request), or
- Trust the DB JOIN (correct but hard to audit)

The `getDeviceHistory` C-2 fix throws `ForbiddenException` when `tenantId` is null — but because `tenantId` is never in the JWT, `CurrentUser('tenantId')` always returns null from the token. It must be pulling from `user.tenantId` on the full user object. Any endpoint that takes `tenantId` from JWT claims silently gets null.

**Fix:** Include `tenantId` in the JWT payload:
```typescript
const token = this.jwtService.sign({
  sub: user.id, email: user.email, role: user.role,
  branchId, tenantId: (user as any).tenantId ?? null,
});
```
Update `jwt.strategy.ts` to extract `tenantId` from the payload and put it on `request.user`.

### BLK-4 · Health endpoint is DB-blind — containers pass health check when database is down

**File:** `backend/src/health.controller.ts`

```typescript
@Get()
check() {
  return { status: 'ok', timestamp: new Date().toISOString() };
}
```

The `/api/v1/health` endpoint returns `200 OK` unconditionally — it does not check database connectivity, Redis, or any application state. The docker-compose healthcheck for the `backend` service uses this endpoint. If PostgreSQL crashes, the backend container still reports healthy and docker-compose never restarts it or blocks the frontend from starting.

**Fix:** Add a real DB probe:
```typescript
@Get()
async check() {
  await this.prisma.$queryRaw`SELECT 1`;
  return { status: 'ok', timestamp: new Date().toISOString() };
}
```
Catch DB errors and return `503` to trigger container restart policies.

### BLK-5 · SUPER_ADMIN_PASSWORD placeholder not validated by env validator

**File:** `backend/.env.production.example:37` / `scripts/validate-prod-env.ps1`

The `.env.production.example` documents:
```
SUPER_ADMIN_PASSWORD=REPLACE_WITH_STRONG_PASSWORD
```

`validate-prod-env.ps1` has 20+ checks for JWT, DB URL, CORS, PORT — but **no check for `SUPER_ADMIN_PASSWORD`**. If the file is copied from the example and `SUPER_ADMIN_PASSWORD` is left as the placeholder, the seed script creates an admin account with a literal placeholder password. This would never be caught before deploy.

**Fix:** Add to `validate-prod-env.ps1`:
```powershell
$sa = $be["SUPER_ADMIN_PASSWORD"]
$saPlaceholder = ($sa -match "REPLACE|CHANGE_ME|placeholder") -or ($sa.Length -lt 12)
Check "BE-13" "SUPER_ADMIN_PASSWORD not a placeholder" (-not $saPlaceholder) `
    "$(if ($saPlaceholder) {'Super admin password is placeholder or too short!'})"
```

### H-1 · Exception filter exposes validation error arrays — schema information leak

**File:** `backend/src/common/filters/http-exception.filter.ts:26-28`

```typescript
if (typeof body === 'object' && body !== null && 'message' in body) {
  message = (body as any).message;   // string | string[]
}
```

NestJS `ValidationPipe` produces `message: ["email must be an email", "password must be longer than 8 characters"]`. This array is passed directly to the client, revealing field names, types, and constraints. An attacker can enumerate the full DTO schema from 400 responses.

**Fix:** Sanitise in production:
```typescript
const isProd = process.env.NODE_ENV === 'production';
if (isProd && status === 400 && Array.isArray(message)) {
  message = 'Validation failed';
}
```

### H-2 · PermissionGuard queries database on every protected request

**File:** `backend/src/common/guards/permission.guard.ts:27-29`

```typescript
const has = await this.prisma.rolePermission.findUnique({
  where: { role_permission: { role: user.role, permission } },
});
```

Every API call to a guarded endpoint hits the database for a permissions lookup. Permissions almost never change at runtime (they're seeded on startup). With a single SUNMI device making ~5 req/s, this adds 5 extra DB queries/s just for permission checks — and scales linearly with concurrent users.

**Fix (minimal):** Cache in-process with a 60-second TTL using a plain `Map`:
```typescript
private cache = new Map<string, number>(); // 'role:permission' → expiresAt

async canActivate(...) {
  const key = `${user.role}:${permission}`;
  if ((this.cache.get(key) ?? 0) > Date.now()) return true;
  const has = await this.prisma.rolePermission.findUnique({ ... });
  if (!has) throw new ForbiddenException(...);
  this.cache.set(key, Date.now() + 60_000);
  return true;
}
```

### H-3 · Audit log writes are silent-fail — audit trail can break undetected

**File:** `backend/src/audit-log/audit-log.service.ts:21-24`

```typescript
async log(entry: AuditLogEntry): Promise<void> {
  try {
    await this.prisma.auditLog.create({ data: entry as any });
  } catch (err) {
    console.warn('[AuditLog] Failed to write audit log:', err);
  }
}
```

If the database is under load, disk space is low, or the audit table is locked, `log()` silently swallows the error. The calling code has no way to know the audit entry was lost. For a financial POS system this is unacceptable — silent audit failures violate compliance requirements.

**Fix:** Elevate failed audit logs to an error metric and optionally queue to disk:
```typescript
} catch (err) {
  this.logger.error('[AuditLog] WRITE FAILED — entry lost', err);
  // Optionally: append to a fallback file so nothing is truly lost
}
```

### H-4 · No off-site backup — data loss if server disk fails

**File:** `scripts/backup-prod.ps1` / `scripts/backup-local.ps1`

`backup-prod.ps1` writes backups to `D:\FixITPro_Backup` — the same physical machine as the production database. A disk failure, power surge, or Windows corruption event destroys both production data and all backups simultaneously.

`backup-local.ps1` has commented-out `CloudDir` and `ExtDrive` parameters that are empty by default.

**Fix:** Before go-live, configure at least one of:
- `$Cfg.CloudDir` → network share or rclone-mounted cloud folder (Google Drive, S3)
- `$Cfg.ExtDrive` → external USB drive with scheduled auto-mount

Minimum: daily off-site sync. For repair-shop data (customer repairs, payments), losing a day is significant.

### H-5 · Rollback does not reverse database migrations

**File:** `scripts/rollback-prod.ps1:3-18` (header comment)

The script header explicitly documents: *"สคริปต์นี้กู้คืน BUILD เท่านั้น — ไม่กู้คืน database โดยอัตโนมัติ"* (This script restores BUILD only — does not restore database automatically).

If a migration adds a NOT NULL column and the rollback restores the old code, the old code crashes on startup because the old Prisma client doesn't know about the new column. The operator must manually choose whether to restore the DB (losing data written after migration) or forward-fix the new code.

This is a **documented limitation** rather than a bug, but there is no runbook or decision tree for the operator in this scenario.

**Fix (pre-launch):** Create `docs/release-readiness/rollback-runbook.md` documenting:
1. Code-only rollback steps (current `rollback-prod.ps1`)
2. DB rollback decision criteria (is the new migration schema-additive only?)
3. Full rollback steps using `pg_restore` from the pre-deploy backup
4. How to verify each scenario succeeded

### H-6 · Content-Security-Policy header missing from nginx

**File:** `nginx/templates/app.conf.template` (omission)

The nginx template adds `X-Frame-Options`, `X-Content-Type-Options`, `Referrer-Policy`, and `Strict-Transport-Security` — but no `Content-Security-Policy` header. Without CSP, the browser does not restrict which scripts, styles, or frames the page can load. An XSS vulnerability in the app or a compromised CDN dependency has no fallback restriction.

**Fix:** Add to both `APP_DOMAIN` and `API_DOMAIN` server blocks:
```nginx
add_header Content-Security-Policy
  "default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline'; img-src 'self' data: blob:; connect-src 'self' ws:; frame-ancestors 'none';"
  always;
```
Start permissive and tighten after verifying no console CSP violations.

### H-7 · `UPLOADS_BASE_DIR` fallback uses `__dirname` — fragile path in production

**File:** `backend/src/main.ts:19`

```typescript
const uploadsBase = process.env.UPLOADS_BASE_DIR || join(__dirname, '..', 'uploads');
```

`__dirname` in the compiled `dist/` output resolves to `D:\FixITPro\backend\dist`. The fallback becomes `D:\FixITPro\backend\dist\..\uploads` = `D:\FixITPro\backend\uploads`. If the uploads directory exists at a different path or the dist layout changes, uploaded repair images become inaccessible silently (no startup error).

**Fix:** Make `UPLOADS_BASE_DIR` required in production:
```typescript
const uploadsBase = process.env.UPLOADS_BASE_DIR;
if (!uploadsBase && isProd) throw new Error('UPLOADS_BASE_DIR must be set in production');
app.useStaticAssets(uploadsBase ?? join(__dirname, '..', 'uploads'), { prefix: '/uploads' });
```
Add `UPLOADS_BASE_DIR` check to `validate-prod-env.ps1`.

### M-1 · Upload size limit too large — disk exhaustion risk

**File:** `nginx/nginx.conf:30`

```nginx
client_max_body_size 20M;
```

A 20 MB limit is appropriate for high-res DSLR photos but excessive for SUNMI camera output (typically 1–3 MB). An operator or attacker can upload 6 files × 20 MB = 120 MB per repair ticket. With 200 repairs/month and careless photo uploads, disk fills within months.

**Fix:** Reduce global limit to `5M`. If large photo uploads are genuinely needed, apply `client_max_body_size 10M` only to the `/api/v1/repairs/*/images` location block.

### M-2 · Certbot renewal failures are silent

**File:** `docker-compose.prod.yml:159`

```yaml
entrypoint: /bin/sh -c "certbot renew --quiet; sleep 12h"
```

`--quiet` suppresses all output including renewal failure messages. TLS certificate expiry goes unnoticed until HTTPS stops working for all clients, including the SUNMI APK.

**Fix:** Remove `--quiet`:
```yaml
entrypoint: /bin/sh -c "certbot renew --no-random-sleep-on-renew 2>&1 | tee /var/log/certbot-renew.log; sleep 12h"
```
Optionally ship the log to the host via a volume mount for monitoring.

### M-3 · Repair upload images cached with `immutable` flag for 30 days

**File:** `nginx/templates/app.conf.template:48-50`

```nginx
location /uploads/ {
    expires 30d;
    add_header Cache-Control "public, immutable";
}
```

`immutable` tells browsers the URL content will **never change**. If a technician re-uploads a corrected repair photo at the same URL (same filename), every browser that cached the original shows the stale image for 30 days with no way to bust the cache.

**Fix:** Remove `immutable`. Use `max-age=86400` (1 day). Image URLs already include random timestamps in the filename (e.g. `1748749823-987654321.jpg`) so cache busting via URL rename is the correct pattern:
```nginx
expires 1d;
add_header Cache-Control "public, max-age=86400";
```

### M-4 · Redis container defined in docker-compose but no backend code uses it

**File:** `docker-compose.prod.yml:31-50`

Redis is declared as a service with password, persistence, and health checks. The `backend` service receives `REDIS_URL`. However, the backend codebase has no Redis client, no `@nestjs/bull`, no `ioredis`, and no cache module. The `REDIS_URL` env var is unused.

Running an unused Redis container wastes ~50 MB RAM and presents an unnecessary attack surface.

**Fix (pre-launch):** Either:
- Remove the Redis service from `docker-compose.prod.yml` until Redis is actually used (recommended), or
- Use it for PermissionGuard caching (H-2 fix) — this would justify its inclusion.

### M-5 · AuditLog ipAddress and userAgent fields always null

**File:** `backend/src/audit-log/audit-log.service.ts:13-14`

The `AuditLogEntry` interface has `ipAddress` and `userAgent` fields, and the Prisma schema stores them. But no middleware or interceptor populates these fields — all call sites pass `actorId` and `actorName` only. The audit log cannot tell which device performed an action.

For a multi-branch POS, knowing whether a sale was made from the SUNMI device or a desktop browser is operationally important.

**Fix:** Add a NestJS interceptor or middleware that extracts IP and user-agent from the request and stores them on the `cls` context, then reads them in `AuditLogService.log()`. Alternatively, pass them explicitly from controller call sites.

### M-6 · Rate limit may be too restrictive during shift-close peak

**File:** `nginx/templates/app.conf.template:76-79`

```nginx
location / {
    limit_req zone=api_general burst=30 nodelay;
```

The `api_general` zone allows `60r/m` (1 req/s avg) with `burst=30`. During shift-close, the SUNMI makes multiple simultaneous requests (close shift, audit log, print summary, fetch notifications). If multiple staff members close shifts simultaneously on a busy day, the burst of 30 is consumed quickly and subsequent requests receive `429`.

**Fix:** Increase rate limit to `120r/m` with `burst=50`. Auth endpoints can remain at `10r/m` for brute-force protection.

### M-7 · HSTS `preload` flag committed to nginx template

**File:** `nginx/templates/app.conf.template:39`

```nginx
add_header Strict-Transport-Security "max-age=63072000; includeSubDomains; preload" always;
```

The `preload` directive submits the domain to browsers' HSTS preload lists. Once a domain is preloaded, it cannot be served over HTTP for years even if TLS is removed. If the shop later moves to a different domain or needs HTTP-only LAN access, they are locked in.

This is only a risk if the actual domain is submitted to the preload list, but the configuration encourages it.

**Fix:** Remove `preload` from the HSTS header for now. Add it back only after confirming the domain is permanent and TLS will always be available:
```nginx
add_header Strict-Transport-Security "max-age=63072000; includeSubDomains" always;
```

### L-1 · JWT expires in 8h with no refresh token — staff must re-login each shift

**File:** `backend/.env.production.example:16`

`JWT_EXPIRES_IN=8h`. An 8-hour shift is fine, but if a shift runs 9 hours or staff leave the device logged in overnight, the session expires. There is no refresh token mechanism. The user sees an opaque "401 Unauthorised" and must re-enter credentials while mid-transaction.

**Fix (post-launch):** Implement a refresh token endpoint with an HTTP-only cookie (30-day expiry). For now, document in operator manual that staff should re-login if they see 401.

### L-3 · bcrypt rounds at 12 — ~250ms login latency

**File:** `backend/src/auth/auth.service.ts:27` (implicit via bcrypt.compare)

bcrypt rounds = 12 (verified from `auth.service.ts` using `bcrypt.compare`). On the production server, this produces ~250ms hash verification per login. For a LAN POS used by 2–5 staff, this is acceptable. For a cloud SaaS with many concurrent logins, this can cause login queue stalls.

**Fix:** No action required for the current single-branch deployment. Re-evaluate if multi-tenant cloud usage grows.

### L-4 · Backup retention logic can delete before 30 days in low-activity periods

**File:** `scripts/backup-local.ps1` (retention section)

The script keeps a minimum of 7 backups and deletes backups older than 30 days. In a period of low backup frequency (e.g., backup scheduled but server was off for 3 weeks), only 7 backups may exist — all older than 30 days — and all 7 would be deleted.

**Fix:** Change logic to `DELETE only if (age > 30 days) AND (totalBackups > 7)` — ensuring at least 7 backups are always retained regardless of age.
